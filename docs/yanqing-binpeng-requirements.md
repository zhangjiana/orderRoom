# 宴请宾朋需求梳理

## 当前状态

- 当前仓库只有微信小程序前端原型。
- 没有服务端、数据库、消息通知、地图检索、商家入驻、客户邀请等真实业务能力。
- 现在的预订数据来自本地存储，只适合演示界面，不具备正式上线条件。

## 产品定位

“宴请宾朋”是一个面向餐厅包间预订场景的小程序，核心包含两类用户：

- 商家端：餐厅老板、店长、前厅经理、接待人员
- 客户端：需要预订包间的个人或企业客户

目标是把“商家开通 -> 建包间 -> 客户找店 -> 浏览包间 -> 提交预订 -> 商家确认 -> 客户通知被邀请人”这条链路打通。

## 角色与权限

### 1. 商家端

商家首次进入小程序，需要完成开通流程：

- 微信手机号授权登录
- 填写商家名称
- 填写餐厅地址
- 地图选点确认经纬度
- 上传门店封面、环境图
- 配置营业时间
- 配置联系电话
- 创建包间

商家后续可以：

- 管理门店资料
- 管理包间
- 管理预订订单
- 确认/拒绝订单
- 设置包间是否可预订
- 查看某个时间段的预订情况

### 2. 客户端

客户进入小程序后：

- 微信手机号授权登录
- 获取定位，查看附近商家
- 也可以搜索商家名称
- 进入商家详情页
- 浏览商家包间
- 选择日期、时间、人数、联系人、手机号、备注
- 提交预订申请
- 接收商家确认结果
- 预订成功后生成邀请函并分享给被邀请人

## 核心业务流程

### 流程 A：商家开通

1. 商家登录
2. 授权手机号
3. 填写门店基础信息
4. 地图选点确认地址
5. 创建包间
6. 开通完成，进入商家工作台

### 流程 B：客户找店并预订

1. 客户登录
2. 授权手机号
3. 获取当前位置
4. 按距离展示附近餐厅
5. 支持搜索商家名称
6. 进入商家详情页
7. 查看包间列表和可预订状态
8. 选择时间、人数、联系人
9. 提交预订申请

### 流程 C：商家确认

1. 商家收到新预订通知
2. 查看预订详情
3. 确认或拒绝
4. 系统通知客户

### 流程 D：客户发邀请函

1. 订单被确认
2. 客户生成邀请函
3. 邀请函包含门店、包间、时间、地图位置
4. 邀请函通过微信分享发送给被邀请人

## 地图功能要求

地图是这个产品的核心能力，不能只是展示地址文本。

必须包含：

- 用户定位
- 附近商家检索
- 商家地图标点
- 商家入驻时地图选点
- 商家详情页导航入口
- 邀请函中显示地图位置

建议使用：

- 微信位置接口
- 腾讯位置服务地图 SDK

## 消息通知要求

至少包含两类通知：

- 商家收到新预订通知
- 客户收到确认/拒绝通知

建议使用：

- 微信订阅消息

## 技术建议

### 方案建议

当前已明确不使用微信云开发，改为自建独立后台服务。

原因：

- 避免云开发成本和平台绑定
- 后续更容易接入独立数据库、对象存储、消息系统
- 更适合同时承载小程序端、商家端、admin 端

### 推荐技术栈

- 小程序前端：原生微信小程序
- 地图：腾讯位置服务 + 微信地图能力
- 后端：Node.js + Express
- 数据库：前期可用本地 JSON / SQLite，后续迁移 MySQL 或 PostgreSQL
- 文件存储：本地 / OSS / COS
- 消息：微信订阅消息
- Admin 端：React + Vite

## 对当前原型的结论

当前版本只能算“视觉原型”，不能算“可用产品”。

如果继续做，我建议下一步不要再只改样式，而是直接重构为：

1. 商家端工作台
2. 客户端找店页
3. 商家详情与包间列表
4. 预订提交流程
5. 云开发后端和地图能力
